## What Was Built

### 1. Search Framework

**Files:** `app/Core/SearchQuery.php`, `app/Core/Paginator.php`

Reusable across every module. `SearchQuery::fromRequest($_GET, $allowedSorts)` builds a sanitised value object from query parameters: page, limit (max 100), sort column (whitelisted), direction, free-text search, and arbitrary filters. `Paginator` does pure pagination math (total, pages, offset, has_next/prev). Every repository search method returns `['data' => [...], 'paginator' => Paginator]`.

This eliminated repeated pagination/filter logic across Assets, IOCs, Threats, and Correlations.

---

### 2. Asset Management Module

**Files:**
- `app/Domain/Asset/AssetEntity.php` (extended: `operating_system`, `network_segment`, `notes`)
- `app/Domain/Asset/AssetRepository.php` (extended: `search()`, linked-record queries, notes)
- `app/Controllers/AssetController.php`
- `app/Views/pages/assets/{index,detail,create,edit}.php`

Full CRUD with search, filters (criticality, type, status, department), sort, pagination. Detail page shows linked Alerts, Incidents, Correlations, and Threats (joined via source feed / asset_id), plus an analyst notes thread. Create/Edit/Delete require `analyst` role minimum; viewers get read-only list and detail access.

---

### 3. IOC Management Module

**Files:**
- `app/Domain/IOC/IOCEntity.php` (extended: `false_positive`, `expiry_at`, `tags`)
- `app/Domain/IOC/IOCRepository.php` (extended: `search()`, history, false-positive toggle)
- `app/Controllers/IOCController.php`
- `app/Views/pages/iocs/{index,detail,create,edit}.php`

Manual IOC creation defaults to `source = 'manual'`, `confidence_score = 75`. Every create/update/flag action writes to `ioc_history` for a full audit trail, shown as a timeline on the detail page. False-positive flag is a one-click toggle. Detail page shows linked correlations, alerts, and related threats (joined via source feed). Duplicate value detection prevents re-adding existing IOCs.

---

### 4. Threat Intelligence Explorer

**Files:**
- `app/Domain/Threat/ThreatEntity.php` (extended: `cve_id`, `affected_systems`)
- `app/Domain/Threat/ThreatRepository.php` (extended: `search()`, CVE/date-range filters, linked queries)
- `app/Controllers/ThreatController.php`
- `app/Views/pages/threats/{index,detail}.php`

**Read-only** — threats are ingested by Workers only (URLHaus, OTX, NVD, CISA), never created manually. Search by title, description, or CVE ID. Filters: severity, source feed, MITRE technique, date range. Detail page resolves the MITRE technique via `MitreMappingService::resolveId()` and shows related IOCs and correlations linked through the shared source feed.

---

### 5. Correlation Explorer

**Files:**
- `app/Repositories/CorrelationRepository.php`
- `app/Controllers/CorrelationController.php`
- `app/Views/pages/correlations/{index,detail}.php`

**Read-only** — correlations are generated by the Intelligence Worker only. Search by IOC value, asset name, or explanation text. Filters: type, severity, confidence threshold, date range. Detail page resolves and displays the actual linked IOC/Asset/Alert/Incident records (not just IDs), plus a single-event timeline.

---

### 6. Role-Aware Dashboard

**Files:**
- `app/Controllers/DashboardController.php` (rewritten — routes by role via `match()`)
- `app/Views/pages/dashboard/{admin,analyst,viewer}.php`

| Role | Dashboard shows |
|------|-----------------|
| Admin | System status, log events, security score, worker/feed health, recent activity, cyber object counts with links, user management shortcut |
| Analyst | Assigned alerts/incidents (falls back to all-open if none assigned), high-risk critical assets, latest threats, recent correlations, IOC feed, security score |
| Viewer | Security score, asset/IOC/threat counts, MITRE distribution, latest threats, recent correlations — fully read-only |

`DashboardController::getAssignedOrOpen()` implements the fallback logic agreed in the approval: if an analyst has no assignments, the dashboard shows all open alerts/incidents instead of an empty state.

---

### 7. Timeline & Pagination Components

**Files:** `app/Views/components/timeline.php`, `app/Views/components/pagination.php`

Reusable partials called via `$view->partial('timeline', [...])` and `$view->partial('pagination', [...])`. Used on the Correlation detail page (timeline) and on Assets/IOCs/Threats/Correlations index pages (pagination).

**Bug fixed during this build:** `View::render()` previously only injected `$view` into the layout scope, not into page templates. Since multiple new pages call `$view->partial()` directly, this would have caused undefined-variable errors. Fixed by moving `$view = $this;` before the page template is required, so it's available in both page and layout scope. This is the only change to `View.php` — one line moved, nothing removed.

## Database Migration

**File:** `database/migrations/008_soc_operations.sql`

Fully idempotent — uses `ADD COLUMN IF NOT EXISTS`, `ADD INDEX IF NOT EXISTS`, `CREATE TABLE IF NOT EXISTS`, and a guarded `INFORMATION_SCHEMA` check before adding the `resolved_by` foreign key (since MySQL has no native `ADD CONSTRAINT IF NOT EXISTS`). Safe to re-run.

**New tables:**

| Table | Purpose |
|-------|---------|
| `asset_notes` | Analyst notes on assets |
| `ioc_history` | IOC event/change log |

**New columns:**

| Table | Columns |
|-------|---------|
| `assets` | `operating_system`, `network_segment`, `notes` |
| `iocs` | `false_positive`, `expiry_at`, `tags` (JSON) |
| `threats` | `cve_id`, `affected_systems` |
| `correlations` | `resolved_at`, `resolved_by` (FK → users) |

No existing data is modified or deleted.

## Architecture Compliance

- ✅ No SQL in Controllers — all queries in Repositories
- ✅ No business logic in Views — only presentation and light formatting
- ✅ Search framework is pure value objects, no SQL
- ✅ CSRF validated on every state-changing POST
- ✅ XSS protection via `Security::e()` on all output
- ✅ Threats and Correlations remain worker-generated only — no manual create/edit/delete routes exist for them
- ✅ No Composer, no frameworks, shared hosting compatible
- ✅ Backward compatible with v0.1.0–v0.5.0
